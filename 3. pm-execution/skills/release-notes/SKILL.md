# 출시 노트 생성기

기술 티켓, PRD 또는 내부 변경 로그를 세련된 사용자용 출시 노트로 변환합니다.

### 맥락

당신은 **$ARGUMENTS**를 위한 출시 노트를 작성하고 있습니다.

사용자가 파일(JIRA 내보내기, Linear 티켓, PRD, Git 로그 또는 내부 변경 로그)을 제공하면 먼저 읽으십시오. 제품 URL이 언급되면 웹 검색을 사용하여 제품과 대상을 파악하십시오.

### 지침

1. **원천 자료 수집**: 제공된 모든 티켓, 변경 로그 또는 설명을 읽으십시오. 다음 내용을 추출합니다:
   - 무엇이 변경되었는가 (기능, 개선 또는 수정)
   - 누구에게 영향을 미치는가 (어떤 사용자 세그먼트)
   - 왜 중요한가 (사용자 이점)

2. **변경 사항 분류**:
   - **신규 기능 (New Features)**: 완전히 새로운 기능
   - **개선 사항 (Improvements)**: 기존 기능의 강화
   - **버그 수정 (Bug Fixes)**: 해결된 문제
   - **중요한 변경 (Breaking Changes)**: 사용자 조치가 필요한 사항 (마이그레이션, API 변경 등)
   - **지원 종료 (Deprecations)**: 곧 사라질 기능

3. **각 항목 작성 원칙**:
   - 기술적 변경이 아닌 사용자 이점을 먼저 내세우십시오.
   - 전문용어, 내부 코드명, 티켓 번호는 피하고 쉬운 언어를 사용하십시오.
   - 각 항목은 1-3문장으로 유지하십시오.
   - 사용자가 이미지를 제공했다면 포함하십시오.

   **변경 예시**:
   - 기술적: "대시보드 API 엔드포인트에 Redis 캐싱 레이어 구현"
   - 사용자용: "대시보드 로딩 속도가 최대 3배 빨라졌습니다. 이제 기다리는 시간은 줄이고 분석에 더 집중하세요."

   - 기술적: "동시 결제 흐름의 레이스 컨디션 수정"
   - 사용자용: "트래픽이 몰리는 시간대에 일부 주문이 실패할 수 있던 문제를 수정했습니다."

4. **출시 노트 구조화**:

   ```
   # [제품명] — [버전 / 날짜]

   ## 신규 기능
   - **[기능 이름]**: [어떤 기능을 하는지, 왜 중요한지 1-2문장 설명]

   ## 개선 사항
   - **[영역]**: [무엇이 좋아졌고 어떻게 도움이 되는지]

   ## 버그 수정
   - 수정됨: [사용자 관점의 문제 설명]

   ## 중요한 변경 (있는 경우)
   - **필요한 조치**: [사용자가 해야 할 일]
   ```

5. **출시 노트 저장**: 제품의 보이스에 맞게 조정하십시오. (B2B는 전문적, 소비재는 친근함, API는 개발자 중심)

마크다운 문서로 저장하십시오. 사용자가 HTML 등의 다른 형식을 원한다면 적절히 변환하십시오.
